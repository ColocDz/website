Reusable Pattern — the mental model
• auth() in Server Components = equivalent to the old session check on the server.
• useSession() in Client Components = reactive, updates when user signs in/out.

Fix 3 — Middleware (Protect Routes Server-Side)
  This runs before ANY page renders — users cannot reach protected pages without logging in

Create ONE file at the root of your project. This replaces every manual auth check inside page components.

middleware.ts  (root)
// middleware.ts  ← create at the ROOT (same level as package.json)
export { auth as middleware } from '@/auth';
 
export const config = {
  matcher: [
    // These routes require login — anyone not logged in
    // is automatically redirected to /login
    '/profile/:path*',
    '/messages/:path*',
    '/adding-post/:path*',
    '/settings/:path*',
    // Add any future protected route here
  ],
};

You also need to update auth.ts to export the middleware-compatible function. Add this to your auth.ts:
auth.ts — add callbacks
// auth.ts — add this to your NextAuth config:
export const { handlers, signIn, signOut, auth } = NextAuth({
  // ...your existing config...
 
  callbacks: {
    // This runs on every request caught by middleware
    authorized({ auth, request: { nextUrl } }) {
      const isLoggedIn = !!auth?.user;
      const isProtected = nextUrl.pathname.startsWith('/profile')
        || nextUrl.pathname.startsWith('/messages')
        || nextUrl.pathname.startsWith('/adding-post')
        || nextUrl.pathname.startsWith('/settings');
 
      if (isProtected && !isLoggedIn) {
        // Redirect to login, then back to the page they wanted
        const redirectUrl = new URL('/login', nextUrl.origin);
        redirectUrl.searchParams.set('callbackUrl', nextUrl.href);
        return Response.redirect(redirectUrl);
      }
      return true;
    },
  },
});

Reusable Pattern — adding a new protected route
• To protect any new page, just add it to the matcher array in middleware.ts.
• You NEVER need to add auth checks inside the page component itself.
• The format '/settings/:path*' means /settings AND /settings/anything.
• Public routes (/, /login, /signup, /posts) don't need to be listed anywhere.



Fix 4 — Images: priority, sizes, and blur
  priority vs loading='eager' — these are NOT the same. Use priority for above-fold images.

The difference, clearly:
Prop	What it actually does
loading='eager'	Disables lazy loading. That's all. Image still competes with other resources.
priority	Disables lazy loading AND adds <link rel='preload'> in the HTML head — browser fetches it before anything else renders. Also suppresses Next.js LCP warning.
sizes (string)	Tells the browser which image size to pick at each screen width. Next.js uses this to auto-generate srcSet — never set srcSet manually.
placeholder='blur'	Shows a blurred version while the real image loads. Requires blurDataURL for remote images.

The correct pattern for a posts grid:
posts/page.tsx
// posts/page.tsx or app/page.tsx
 
// Rule:
// - First 1-3 visible cards → priority={true}
// - Everything else        → nothing (lazy by default)
 
{filteredPosts.map((post, index) => (
  <div key={post.id} className='group bg-white border rounded-lg overflow-hidden'>
    <div className='relative h-48'>  {/* ← parent MUST have a defined height */}
      <Image
        src={post.image}
        alt={post.title}
        fill                        // fills the parent div
        priority={index < 3}        // true for first 3, false for rest
        sizes='(max-width: 640px) 100vw,
               (max-width: 1024px) 50vw,
               33vw'
        // ↑ Next.js generates srcSet automatically from this.
        // Do NOT add srcSet manually.
        className='object-cover group-hover:scale-105 transition-transform duration-300'
      />
    </div>
    {/* rest of card */}
  </div>
))};

The correct pattern for user avatars:
Avatar pattern
// messages/page.tsx, profile/page.tsx
// Avatars are small, below the hero — always lazy (default), no priority
 
<div className='relative w-10 h-10'>  {/* defined size on parent */}
  <Image
    src={conversation.avatar}
    alt={conversation.name}
    fill
    sizes='40px'                     // just the display size, no responsive needed
    className='rounded-full object-cover'
    // No priority, no loading='eager' — lazy by default is correct here

// lib/get-blur.ts
import { getPlaiceholder } from 'plaiceholder';
 
export async function getBlurDataURL(src: string) {
  const buffer = await fetch(src).then(r => r.arrayBuffer());
  const { base64 } = await getPlaiceholder(Buffer.from(buffer));
  return base64;
}
 
// Use it in a Server Component:
// const blurDataURL = await getBlurDataURL(post.imageUrl);

Reusable Pattern — image checklist for any new page
• 1. Is the image above the fold (visible immediately)? → priority={index < 3}
• 2. Is it in a grid or below fold? → nothing (lazy is default)
• 3. Does the parent div have a defined height? → always, when using fill
• 4. Set sizes string to match your Tailwind grid (100vw mobile, 50vw tablet, 33vw desktop)
• 5. Never set srcSet manually in Next.js — sizes does it for you
• 6. Never use loading='eager' — use priority instead

export async function POST(req: Request) {
  const session = await auth();
  if (!session) return new Response('Unauthorized', { status: 401 });
 
  const body = await req.json();
 
  // Validate with the SAME schema used in the form — one source of truth
  const parsed = createPostSchema.safeParse(body);
  if (!parsed.success) {
    return Response.json({ errors: parsed.error.flatten() }, { status: 400 });
  }
 
  // parsed.data is fully typed here
  // const post = await db.post.create({ data: { ...parsed.data, userId: session.user.id } });
  return Response.json({ success: true });
}

Reusable Pattern — every form from now on
• 1. Create schema in lib/validations/[entity].ts
• 2. Use zodResolver in useForm — no manual validation state needed
• 3. Register fields with {...register('fieldName')} — no value/onChange
• 4. Show errors with {errors.fieldName?.message}
• 5. Import the SAME schema in the API route to validate the request body
• One schema = client validation + server validation + TypeScript types. Never duplicated.

      || post.description.toLowerCase().includes(searchQuery.toLowerCase())

    const matchesType   = !selectedType   || post.type   === selectedType;

Reusable Pattern — when to use useMemo
• Use useMemo when: you transform or filter a list, and the component has other state that changes.
• Use useCallback when: you pass a function as a prop to a child component.
• Do NOT useMemo everything — it has overhead. Only memo expensive or frequently-called computations.
• Rule of thumb: if the function loops over an array, wrap it in useMemo.



Fix 10 — Server Components: Move Data Fetching Off the Client
  This is the biggest architectural win. Replaces hardcoded data with real DB queries when ready.

You don't need a database yet to benefit from this. Start structuring your pages correctly now so adding the DB is just swapping one line.

// components/posts-grid.tsx — the 'use client' interactive part
'use client';
import { useState, useMemo } from 'react';
import Image from 'next/image';
import type { Post } from '@/types';
 
interface PostsGridProps {

}
 
export function PostsGrid({ initialPosts }: PostsGridProps) {
  const [searchQuery,   setSearchQuery]   = useState('');
  const [selectedType,  setSelectedType]  = useState('');
  const [selectedWilaya,setSelectedWilaya]= useState('');
 
  const filteredPosts = useMemo(() =>
    initialPosts.filter(post => {

      const matchesType   = !selectedType   || post.type   === selectedType;

                src={post.imageUrl}

                fill

Reusable Pattern — the Server/Client split rule
• A component needs 'use client' ONLY if it uses: useState, useEffect, useRef, onClick, onChange, or any browser API.
• Everything else should be a Server Component — no directive needed.
• Data fetching always belongs in Server Components (or API routes called by Server Components).
• Props flow down: Server fetches data → passes to Client component as props.
• This is the correct Next.js 14 App Router architecture. It is not optional — it is the design.
